Flow Sandbox provides powerful features for developing, testing, and debugging Blazing Flow workflows in isolation.

## Core Features

### 1. Isolated Execution Environment

Run workflows in complete isolation from production systems.

**Benefits:**
- Test breaking changes safely
- Experiment without risk
- Develop independently
- Validate before production

**Configuration:**

```yaml
sandbox:
  isolation:
    level: strict  # strict, moderate, permissive

    # Resource isolation
    resources:
      cpu: 2
      memory: 4GB
      storage: 10GB

    # Network isolation
    network:
      mode: isolated
      allowed_endpoints:
        - https://api.example.com
        - https://external-service.com
```

### 2. State Management

Manage workflow state independently from production.

**Storage Options:**

```yaml
sandbox:
  state:
    # In-memory (fastest, non-persistent)
    storage: memory

    # PostgreSQL (persistent, queryable)
    # storage: postgres
    # connection: postgresql://localhost/sandbox

    # Redis (fast, persistent)
    # storage: redis
    # connection: redis://localhost:6379

    # TTL for automatic cleanup
    ttl: 24h
```

**State Operations:**

```bash
# View current state
blazing sandbox state get <execution-id>

# Modify state for testing
blazing sandbox state set <execution-id> --key foo --value bar

# Reset state
blazing sandbox state reset <execution-id>

# Export state snapshot
blazing sandbox state export --output state.json
```

### 3. Event Replay

Replay historical events to test workflows.

**Capture Events:**

```bash
# Capture production events
blazing events capture \
  --source production \
  --output events.json \
  --duration 1h
```

**Replay in Sandbox:**

```bash
# Replay captured events
blazing sandbox replay events.json \
  --sandbox my-sandbox \
  --speed 10x  # Replay at 10x speed
```

**Selective Replay:**

```yaml
sandbox:
  replay:
    # Filter events
    filters:
      - event_type: "order.created"
      - user_id: "test-user-123"

    # Time manipulation
    speed: 10x          # 10x faster
    start_time: "2024-01-01T00:00:00Z"

    # Error injection
    inject_errors:
      - rate: 0.05      # 5% error rate
        type: timeout
      - rate: 0.02      # 2% error rate
        type: network
```

### 4. Time Travel Debugging

Step through workflow execution like a debugger.

**Features:**
- Pause execution at any point
- Step forward/backward through tasks
- Inspect state at each step
- Modify variables and resume

**Usage:**

```bash
# Start workflow in debug mode
blazing sandbox debug test-workflow \
  --sandbox my-sandbox \
  --breakpoint process_data

# Debug commands
> step           # Execute next task
> continue       # Continue to next breakpoint
> back           # Step backward
> state          # Show current state
> set key=value  # Modify state
> restart        # Restart from beginning
```

**Programmatic Debugging:**

```python
from blazing.testing import SandboxDebugger

async def debug_workflow():
    debugger = SandboxDebugger("my-sandbox")

    # Set breakpoints
    debugger.breakpoint("process_data")
    debugger.breakpoint("send_notification")

    # Start execution
    execution = await debugger.run("test-workflow")

    # Wait for breakpoint
    await debugger.wait_for_break()

    # Inspect state
    state = debugger.get_state()
    print(f"Current state: {state}")

    # Modify variables
    debugger.set_variable("retry_count", 5)

    # Continue
    await debugger.continue_execution()
```

### 6. Mock External Services

Mock external APIs and services for testing.

**Define Mocks:**

```yaml
sandbox:
  mocks:
    # Mock HTTP endpoints
    - endpoint: "https://api.stripe.com/*"
      responses:
        - method: POST
          path: "/v1/charges"
          status: 200
          body:
            id: "ch_mock123"
            status: "succeeded"
          latency: 100ms

    # Mock with dynamic responses
    - endpoint: "https://api.example.com/*"
      handler: "./mocks/api-handler.py"

    # Mock failures
    - endpoint: "https://flaky-api.com/*"
      responses:
        - status: 500
          probability: 0.2  # 20% failure rate
        - status: 200
          probability: 0.8  # 80% success rate
```

**Mock Handler:**

```python
# mocks/api-handler.py
from blazing.mocks import MockHandler

handler = MockHandler()

@handler.route("POST", "/api/users")
async def create_user(request):
    """Mock user creation"""
    user_data = request.json()

    return {
        "id": f"mock-{uuid4()}",
        "email": user_data["email"],
        "created_at": datetime.now().isoformat()
    }, 201

@handler.route("GET", "/api/users/:id")
async def get_user(request, id):
    """Mock user retrieval"""
    return {
        "id": id,
        "email": f"user-{id}@example.com"
    }
```

### 7. Chaos Engineering

Inject failures to test resilience.

**Chaos Configuration:**

```yaml
sandbox:
  chaos:
    enabled: true

    # Failure injection
    failures:
      - type: task_timeout
        targets:
          - task: process_data
        rate: 0.1  # 10% of executions

      - type: network_latency
        targets:
          - service: external-api
        latency: 2s-5s  # Random latency between 2-5s

      - type: data_corruption
        targets:
          - state: user_data
        rate: 0.05

      - type: resource_exhaustion
        targets:
          - resource: memory
        threshold: 90%  # Exhaust memory to 90%
```

**Chaos Commands:**

```bash
# Run with chaos
blazing sandbox chaos test-workflow \
  --sandbox my-sandbox \
  --failure-rate 0.1

# Specific chaos scenarios
blazing sandbox chaos test-workflow \
  --inject timeout \
  --inject network-partition \
  --inject cpu-spike
```

### 8. Snapshot & Restore

Save and restore sandbox state for reproducible testing.

**Create Snapshot:**

```bash
# Create snapshot
blazing sandbox snapshot create my-sandbox \
  --name before-migration \
  --description "State before database migration"
```

**Restore Snapshot:**

```bash
# Restore to previous state
blazing sandbox snapshot restore my-sandbox \
  --snapshot before-migration
```

**Snapshot Management:**

```bash
# List snapshots
blazing sandbox snapshot list my-sandbox

# Compare snapshots
blazing sandbox snapshot diff \
  snapshot-1 snapshot-2

# Export snapshot
blazing sandbox snapshot export \
  before-migration \
  --output snapshot.tar.gz
```

### 9. Collaborative Testing

Share sandboxes with team members.

**Share Sandbox:**

```bash
# Share with team member
blazing sandbox share my-sandbox \
  --email teammate@example.com \
  --role developer

# Create shareable link
blazing sandbox link my-sandbox \
  --expires 7d
```

**Access Control:**

```yaml
sandbox:
  access:
    # Team members
    members:
      - email: developer@example.com
        role: developer
      - email: qa@example.com
        role: tester

    # Roles and permissions
    roles:
      developer:
        - deploy
        - execute
        - debug
      tester:
        - execute
        - view_logs
```

### 10. Metrics & Observability

Monitor sandbox execution with detailed metrics.

**Built-in Metrics:**

```bash
# View real-time metrics
blazing sandbox metrics my-sandbox --live

# Export metrics
blazing sandbox metrics export \
  --format prometheus \
  --output metrics.txt
```

## Advanced Features

### A/B Testing

Test different workflow versions side-by-side:

```yaml
sandbox:
  ab_testing:
    enabled: true

    variants:
      - name: control
        workflow: workflow-v1.py
        traffic: 50

      - name: experiment
        workflow: workflow-v2.py
        traffic: 50

    metrics:
      - name: success_rate
        goal: maximize
      - name: latency
        goal: minimize
```
