# v1.7 Migration Guide: Multi-Version Python Support

## What Changed

v1.7 adds support for running services on Python 3.11, 3.12, or 3.13. The platform now:

- Auto-detects your Python version at publish time by reading `sys.version_info` from your local Python environment
- Allows explicit version override via the `--python-version` CLI flag
- Routes service invocations to version-matched executors (Docker or Pyodide)
- Maintains version-specific wheels and snapshots for compatibility

## What Happens to Existing Apps (Zero-Touch Upgrade)

Existing apps are automatically assigned Python 3.12 (the system default) when you upgrade to v1.7. This ensures a zero-touch upgrade experience:

- **No action required**: Apps continue to work identically after upgrading
- **Automatic assignment**: The Python 3.12 assignment happens on your next `blazing publish` command
- **Auto-detection behavior**:
  - If you were already running Python 3.12 locally, nothing changes
  - If you were running Python 3.11 or 3.13 locally, auto-detect will set the correct version based on your environment
- **Backward compatibility**: Pre-v1.7 snapshots restore normally with Python 3.12

## How to Check Your Service's Python Version

After publishing, you can verify which Python version your service is using:

**Option 1: Check publish output**
```bash
blazing publish
# Output includes: "Detected Python version: 3.12"
```

**Option 2: Query service info**
```bash
blazing service info <service-name>
# Look for "python_version" field in the output
```

**Option 3: Check Redis directly** (advanced users)
```python
from backend.dao.service import ServiceDAO

service = await ServiceDAO.get_by_name(app_id="your-app-id", name="service-name")
print(service.python_version)  # e.g., "3.12"
```

## How to Change Python Version

You have two options for setting or changing your service's Python version:

### Option 1: Auto-Detect (Recommended)

Switch your local Python environment, then republish:

```bash
# Using pyenv (recommended)
pyenv local 3.13
blazing publish

# Using conda
conda activate py313-env
blazing publish

# Using venv
source ~/envs/py313/bin/activate
blazing publish
```

The platform reads `sys.version_info` and automatically assigns the correct version.

### Option 2: Explicit Flag Override

Override auto-detection with the `--python-version` flag:

```bash
blazing publish --python-version 3.13
```

This is useful when:
- Your local Python differs from your desired deployment version
- You want to test your service on a different Python version
- You're in a CI/CD pipeline with a fixed Python environment

### What Happens After Changing Version

When you change your service's Python version:

1. **Wheels are rebuilt**: Native wheels (with C extensions) are rebuilt for the new Python version using version-matched build containers
2. **Pure-Python wheels are shared**: Wheels without C extensions are reused across all versions (no rebuild needed)
3. **Snapshots are automatically invalidated**: Existing snapshots are marked invalid because dill serialization is not compatible across Python minor versions
4. **Routing changes**: The service is routed to a version-matched executor:
   - Docker executor: Uses pyenv to activate the correct Python version at runtime
   - Pyodide executor: Routes to the version-specific Pyodide container (e.g., port 8013 for Python 3.13)
5. **First invocation is a cold start**: Because snapshots were invalidated, the first invocation after version change will take longer (full environment setup)

## Snapshot Behavior on Version Change

Blazing uses [dill](https://github.com/uqfoundation/dill) for Python environment serialization. Dill snapshots are **not portable across Python minor versions** (e.g., 3.12 → 3.13).

To prevent deserialization failures, v1.7 implements version-aware snapshot handling:

### Snapshot Tagging (SNAP-01)
Every snapshot is tagged with the `python_version` it was created with:
```python
snapshot_metadata = {
    "python_version": "3.12",
    "app_id": "...",
    "service_name": "...",
    # ...
}
```

### Version Mismatch Detection (SNAP-02)
At executor startup, if a snapshot's `python_version` doesn't match the executor's Python version, the snapshot is rejected and the executor falls back to a cold start:
```python
# Executor checks:
if snapshot.python_version != executor.python_version:
    logger.warning(f"Snapshot version mismatch: {snapshot.python_version} != {executor.python_version}")
    return None  # Cold start fallback
```

### Automatic Invalidation on Version Change (SNAP-03)
When you change a service's Python version via `blazing publish --python-version X.Y`, the platform automatically invalidates all existing snapshots for that service:
```python
# Publish pipeline invalidates snapshots when version changes
if new_version != old_version:
    await snapshot_dao.invalidate_all(app_id=app_id, service_name=service_name)
```

This prevents the executor from attempting to restore incompatible snapshots.

## Executor Routing

v1.7 introduces version-aware executor routing to ensure services run on the correct Python version:

### Docker Executor: Runtime Version Switching

The Docker executor uses a single container with [pyenv](https://github.com/pyenv/pyenv) to support all three Python versions:

- **Installed versions**: 3.11.9, 3.12.7, 3.13.1 (via pyenv)
- **Default version**: 3.12.7 (set via `pyenv global`)
- **Runtime activation**: Uses `PYENV_VERSION` environment variable to activate the correct version before code execution
- **Snapshot support**: Yes (version-validated)
- **Native wheels**: Yes (built with version-matched containers)

### Pyodide Executor: Container-Per-Version

The Pyodide executor uses separate containers for each Python version:

| Python Version | Pyodide Version | npm Package | Container Port |
|----------------|----------------|-------------|----------------|
| 3.11 | 0.25.1 | pyodide@0.25.1 | 8011 |
| 3.12 | 0.27.7 | pyodide@0.27.7 | 8012 |
| 3.13 | 0.28.3 | pyodide@0.28.3 | 8013 |

Each Pyodide container is compiled for a specific Python version. Cross-version usage is not supported because Pyodide's WASM runtime is tied to the Python version it was compiled with.

### Routing Logic

The coordinator routes service invocations based on `service.python_version`:

1. **Query ExecutorRegistry** for an executor matching `python_version` and `executor_type` (Docker or Pyodide)
2. **Fallback to Docker executor** if no Pyodide executor is available for the requested version
3. **Ultimate fallback** to static `PYODIDE_EXECUTOR_URL` environment variable if the registry is empty

This ensures graceful degradation when version-specific executors are unavailable.

## FAQ

### Do I need to republish to use v1.7 features?

**Yes.** Your next `blazing publish` command will auto-detect your Python version and store it in the service metadata. Until you republish, your service continues running on the system default (Python 3.12).

### Can I run different services on different Python versions?

**Yes.** Each service has its own `python_version` field. You can have one service running on Python 3.11, another on 3.12, and another on 3.13 within the same app.

### What happens if I publish from Python 3.10?

**Validation error.** The minimum supported Python version is 3.11. Publishing from Python 3.10 (or earlier) will fail with:
```
ValidationError: Python version must be >= 3.11 (got 3.10)
```

Upgrade your local Python to 3.11+ before publishing.

### Will my snapshots break?

**They are automatically invalidated if the version changes.** When you change a service's Python version (e.g., 3.12 → 3.13), the platform invalidates all existing snapshots to prevent deserialization failures. The first invocation after the version change will be a cold start.

If you keep the same version, snapshots continue working normally.

### Is there downtime during migration?

**No.** Existing services continue running until you republish. The upgrade to v1.7 does not automatically restart or redeploy services. Services are only redeployed when you run `blazing publish`.

### What if I need a Python version not listed (e.g., 3.14)?

**Not supported yet.** v1.7 supports Python 3.11, 3.12, and 3.13 only. Support for future Python versions will be added in subsequent releases. Check the [compatibility matrix](../references/python-version-compatibility-matrix.md) for the latest supported versions.

### How do I know which Pyodide version my service is using?

**Query the executor's health endpoint.** Each Pyodide executor reports its Python version and Pyodide version:
```bash
curl http://localhost:8013/health
# Response: {"status": "healthy", "python_version": "3.13", "pyodide_version": "0.28.3"}
```

You can also check `docker-compose.yml` for the `PYODIDE_VERSION` build argument.

### Can I use Python 3.11 with Pyodide 0.28.3?

**No.** Each Pyodide version is compiled for a specific Python version. The mapping is fixed:
- Python 3.11 = Pyodide 0.25.1
- Python 3.12 = Pyodide 0.27.7
- Python 3.13 = Pyodide 0.28.3

Cross-version usage will result in runtime errors.
