Services are the core building blocks of your application in Blazing Core Compose. Each service represents a containerized workload with its own configuration, resources, and scaling policies.

## Service Definition

A service is defined under the `services` block:

```yaml
version: "v0.2"

services:
  api:                      # Service name
    image: myapp:latest     # Container image
    cpu: 2                  # CPU cores
    memory: 4Gi             # Memory allocation
```

## Minimal Service

The smallest viable service configuration:

```yaml
services:
  web:
    image: nginx:latest
    cpu: 1
    memory: 2Gi
```

This creates a single replica with fixed resources.

## Complete Service Anatomy

Here's a fully configured service with all available options:

```yaml
services:
  api:
    # Container configuration
    image: registry.example.com/api:v2.1.0
    command: ["./api-server"]
    args: ["--port=8080", "--workers=4"]

    # Resource allocation
    cpu: 2
    memory: 4Gi

    # Environment and secrets
    variables:
      DATABASE_URL: postgres://db.example.com/mydb
      LOG_LEVEL: info
      ENVIRONMENT: production

    secrets:
      API_KEY: "!ref:secrets.api-key"
      DB_PASSWORD: "!ref:secrets.db-password"

    # Storage
    storage:
      - path: /data
        size: 10Gi
        type: persistent

    # GPU configuration
    gpu:
      count: 1
      model: nvidia-a100
      sharing:
        mode: exclusive

    # Autoscaling
    autoscaling:
      enabled: true
      min_replicas: 3
      max_replicas: 20
      target_cpu_percent: 70
      target_memory_percent: 80

    # Networking
    network:
      ingress:
        expose:
          - port: 443
            protocol: HTTPS
            tls:
              auto: true
      egress:
        lock_country: true

    # Placement constraints
    placement:
      residency:
        require_countries: ["US", "CA"]
      confidential_compute: preferred

    # Observability
    observability:
      metrics:
        endpoint: /metrics
        format: prometheus
      logs:
        stdout: true
        structured: json

    # Governance
    controls: [PCI-DSS-4.0, DATA-RESIDENCY-NA]
    profiles: [high-availability, cost-optimized]
```

## Field Reference

### Required Fields

| Field | Type | Description | Example |
|-------|------|-------------|---------|
| `image` | string | Container image (registry/name:tag) | `nginx:latest` |
| `cpu` | number | CPU cores (can be fractional) | `2` or `0.5` |
| `memory` | string | Memory allocation with unit | `4Gi` or `512Mi` |

### Container Configuration

#### Image

```yaml
services:
  api:
    image: registry.example.com/api:v2.1.0
```

Supported registries:
- Docker Hub: `nginx:latest`
- GitHub Container Registry: `ghcr.io/user/image:tag`
- Google Container Registry: `gcr.io/project/image:tag`
- Private registries: `registry.example.com/image:tag`

#### Command and Args

Override container entrypoint and arguments:

```yaml
services:
  api:
    image: myapp:latest
    command: ["./custom-entrypoint.sh"]
    args: ["--port=8080", "--debug"]
```

**Note**: `command` replaces ENTRYPOINT, `args` replaces CMD.

### Resource Allocation

#### CPU

Specify CPU cores (fractional or integer):

```yaml
services:
  # Small service
  worker:
    cpu: 0.5              # Half a core

  # Medium service
  api:
    cpu: 2                # 2 cores

  # Large service
  batch:
    cpu: 8                # 8 cores
```

**Cost Impact**: Linear scaling (2 CPU = 2x cost of 1 CPU)

#### Memory

Specify memory with units:

```yaml
services:
  # 512 megabytes
  cache:
    memory: 512Mi

  # 4 gigabytes
  api:
    memory: 4Gi

  # 16 gigabytes
  database:
    memory: 16Gi
```

**Supported Units**: `Mi` (mebibytes), `Gi` (gibibytes)

**Cost Impact**: Linear scaling (4Gi = 2x cost of 2Gi)

### Environment Variables

#### Variables (Plain Text)

```yaml
services:
  api:
    variables:
      DATABASE_URL: postgres://db.example.com/prod
      LOG_LEVEL: info
      ENVIRONMENT: production
      MAX_CONNECTIONS: "100"
```

**Note**: All values are strings (quote numbers).

#### Secrets (Encrypted)

```yaml
services:
  api:
    secrets:
      API_KEY: "!ref:secrets.stripe-api-key"
      DB_PASSWORD: "!ref:secrets.postgres-password"
      JWT_SECRET: "!ref:secrets.jwt-signing-key"
```

Secrets are:
- Encrypted at rest
- Injected as environment variables
- Never logged
- Rotatable via API

**Create secrets via CLI**:

```bash
core secrets create stripe-api-key "sk_live_abc123..."
core secrets create postgres-password "secure-password"
```

## Service Naming

Service names must:
- Be lowercase alphanumeric
- Start with a letter
- Use hyphens for word separation
- Be 1-63 characters long

**Good Names**:
```yaml
services:
  api:           # ✓
  web-frontend:  # ✓
  worker-batch:  # ✓
  db-primary:    # ✓
```

**Bad Names**:
```yaml
services:
  API:           # ✗ Uppercase
  web_frontend:  # ✗ Underscores
  1worker:       # ✗ Starts with number
  very-long-service-name-that-exceeds-the-maximum-length-limit:  # ✗ Too long
```

## Multi-Service Applications

Define multiple services in one deployment:

```yaml
services:
  # Frontend
  web:
    image: myapp/web:latest
    cpu: 1
    memory: 2Gi
    autoscaling:
      min_replicas: 3
      max_replicas: 10

  # Backend API
  api:
    image: myapp/api:latest
    cpu: 2
    memory: 4Gi
    autoscaling:
      min_replicas: 5
      max_replicas: 50

  # Background Workers
  worker:
    image: myapp/worker:latest
    cpu: 4
    memory: 8Gi
    autoscaling:
      min_replicas: 2
      max_replicas: 100

  # Database
  database:
    image: postgres:15
    cpu: 4
    memory: 16Gi
    autoscaling:
      enabled: false    # Fixed size for database
    storage:
      - path: /var/lib/postgresql/data
        size: 100Gi
        type: persistent
```

## Service Communication

Services within the same cluster can communicate via DNS:

```yaml
services:
  api:
    variables:
      DATABASE_URL: postgres://database:5432/mydb
      CACHE_URL: redis://cache:6379
      QUEUE_URL: amqp://queue:5672

  database:
    image: postgres:15

  cache:
    image: redis:7

  queue:
    image: rabbitmq:3
```

**DNS Format**: `{service-name}:{port}`

## Resource Sizing Guide

### CPU Recommendations

| Workload Type | CPU | Example |
|---------------|-----|---------|
| **Static site** | 0.25-0.5 | Simple HTML/JS |
| **API server** | 1-2 | REST/GraphQL API |
| **Web app** | 2-4 | Next.js, React SSR |
| **Worker** | 4-8 | Data processing |
| **Batch job** | 8-16 | ML training, encoding |

### Memory Recommendations

| Workload Type | Memory | Example |
|---------------|--------|---------|
| **Static site** | 512Mi-1Gi | nginx, S3 proxy |
| **API server** | 2Gi-4Gi | Node.js, Go API |
| **Web app** | 4Gi-8Gi | Rails, Django |
| **Worker** | 8Gi-16Gi | Python workers |
| **Database** | 16Gi-32Gi+ | PostgreSQL, MongoDB |

### Right-Sizing Example

Start conservative, scale up based on metrics:

**Phase 1**: Initial deployment

```yaml
services:
  api:
    cpu: 1
    memory: 2Gi
    autoscaling:
      min_replicas: 2
      max_replicas: 10
```

**Phase 2**: After monitoring (CPU at 80%)

```yaml
services:
  api:
    cpu: 2              # Doubled CPU
    memory: 2Gi         # Memory OK
    autoscaling:
      min_replicas: 2
      max_replicas: 10
```

**Phase 3**: After growth (scaling frequently)

```yaml
services:
  api:
    cpu: 2
    memory: 2Gi
    autoscaling:
      min_replicas: 5   # Increased baseline
      max_replicas: 20  # Increased ceiling
```

## Health Checks

### Liveness Probe

Check if container is alive:

```yaml
services:
  api:
    health:
      liveness:
        http:
          path: /health
          port: 8080
        initial_delay_seconds: 30
        period_seconds: 10
        timeout_seconds: 5
        failure_threshold: 3
```

**Behavior**: Restart container if checks fail.

### Readiness Probe

Check if container is ready for traffic:

```yaml
services:
  api:
    health:
      readiness:
        http:
          path: /ready
          port: 8080
        initial_delay_seconds: 10
        period_seconds: 5
        timeout_seconds: 3
        failure_threshold: 3
```

**Behavior**: Remove from load balancer if checks fail.

## Common Patterns

### Pattern 1: Stateless API

```yaml
services:
  api:
    image: myapi:latest
    cpu: 2
    memory: 4Gi
    autoscaling:
      enabled: true
      min_replicas: 3
      max_replicas: 20
    network:
      ingress:
        expose:
          - port: 443
            tls:
              auto: true
```

### Pattern 2: Background Worker

```yaml
services:
  worker:
    image: myworker:latest
    cpu: 4
    memory: 8Gi
    autoscaling:
      enabled: true
      min_replicas: 0      # Scale to zero
      max_replicas: 100
    # No ingress (internal only)
```

### Pattern 3: Database

```yaml
services:
  database:
    image: postgres:15
    cpu: 4
    memory: 16Gi
    autoscaling:
      enabled: false       # Fixed size
    storage:
      - path: /var/lib/postgresql/data
        size: 100Gi
        type: persistent
```

### Pattern 4: GPU Workload

```yaml
services:
  ml-trainer:
    image: ml-training:latest
    cpu: 8
    memory: 32Gi
    gpu:
      count: 2
      model: nvidia-a100
    autoscaling:
      min_replicas: 0
      max_replicas: 10
```
