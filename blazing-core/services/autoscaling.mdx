Autoscaling automatically adjusts the number of service replicas based on demand, optimizing both performance and cost.

## Basic Autoscaling

```yaml
services:
  api:
    image: myapp:latest
    cpu: 2
    memory: 4Gi

    autoscaling:
      enabled: true
      min_replicas: 3
      max_replicas: 20
      target_cpu_percent: 70
```

## Configuration Reference

### Complete Autoscaling Block

```yaml
services:
  api:
    autoscaling:
      # Basic settings
      enabled: true
      min_replicas: 3
      max_replicas: 20

      # CPU-based scaling
      target_cpu_percent: 70

      # Memory-based scaling (optional)
      target_memory_percent: 80

      # Scaling behavior
      scale_up_cooldown_seconds: 45
      scale_down_cooldown_seconds: 300

      # Custom metrics (optional)
      custom_metrics:
        active_connections:
          target: 100
          scale_down_threshold: 30
```

### Field Reference

| Field | Type | Default | Description |
|-------|------|---------|-------------|
| `enabled` | boolean | false | Enable/disable autoscaling |
| `min_replicas` | number | 1 | Minimum replicas (0 for scale-to-zero) |
| `max_replicas` | number | 10 | Maximum replicas (hard ceiling) |
| `target_cpu_percent` | number | 70 | Target CPU utilization (%) |
| `target_memory_percent` | number | none | Optional memory threshold |
| `scale_up_cooldown_seconds` | number | 45 | Wait time before next scale-up |
| `scale_down_cooldown_seconds` | number | 300 | Wait time before scale-down |

## Scaling Strategies

### Strategy 1: Conservative (High Availability)

Keep ample headroom for traffic spikes:

```yaml
autoscaling:
  enabled: true
  min_replicas: 5          # High baseline
  max_replicas: 50
  target_cpu_percent: 50   # Scale at 50% CPU
  scale_up_cooldown_seconds: 30    # Fast scale-up
  scale_down_cooldown_seconds: 600 # Slow scale-down
```

**Use Case**: Production APIs, critical services

**Cost**: Higher baseline ($$$)

### Strategy 2: Balanced (Standard)

Balance cost and performance:

```yaml
autoscaling:
  enabled: true
  min_replicas: 3          # Moderate baseline
  max_replicas: 20
  target_cpu_percent: 70   # Scale at 70% CPU
  scale_up_cooldown_seconds: 45
  scale_down_cooldown_seconds: 300
```

**Use Case**: Most web applications

**Cost**: Moderate ($$)

### Strategy 3: Aggressive (Cost-Optimized)

Minimize replicas, scale reactively:

```yaml
autoscaling:
  enabled: true
  min_replicas: 1          # Minimal baseline
  max_replicas: 20
  target_cpu_percent: 85   # Scale at 85% CPU
  scale_up_cooldown_seconds: 60    # Slower scale-up
  scale_down_cooldown_seconds: 180 # Faster scale-down
```

**Use Case**: Dev/staging, batch jobs

**Cost**: Low ($)

### Strategy 4: Scale-to-Zero (Event-Driven)

No baseline cost when idle:

```yaml
autoscaling:
  enabled: true
  min_replicas: 0          # Scale to zero
  max_replicas: 100
  target_cpu_percent: 70
  scale_up_cooldown_seconds: 0     # Immediate scale-up
  scale_down_cooldown_seconds: 300
```

**Use Case**: Batch processing, webhooks, scheduled jobs

**Cost**: Minimal (pay only when running)

**Note**: Cold starts add 5-15 seconds latency.

## CPU-Based Scaling

### How It Works

1. Monitor average CPU across all replicas
2. If CPU > `target_cpu_percent`: scale up
3. If CPU < `target_cpu_percent`: scale down

### Scaling Formula

```
desired_replicas = current_replicas × (current_cpu / target_cpu)
```

**Example**:
- Current: 5 replicas at 90% CPU
- Target: 70% CPU
- Desired: 5 × (90 / 70) = 6.4 → **7 replicas**

### CPU Target Recommendations

| Target | Behavior | Use Case |
|--------|----------|----------|
| **50%** | Aggressive headroom | Critical services |
| **70%** | Balanced (default) | Standard applications |
| **85%** | Minimal headroom | Cost-sensitive |

**Warning**: Targets >85% risk saturation during scale-up lag.

## Memory-Based Scaling

Optional secondary metric:

```yaml
autoscaling:
  target_cpu_percent: 70
  target_memory_percent: 80
```

**Behavior**: Scale if **either** CPU or memory exceeds target.

### When to Use Memory Scaling

✅ **Use memory scaling when**:
- Application is memory-intensive (caching, data processing)
- Memory leaks are possible
- OOM kills would be catastrophic

❌ **Skip memory scaling when**:
- Application memory is stable
- CPU is the bottleneck
- Simplicity is preferred

## Cooldown Periods

### Scale-Up Cooldown

Wait time before **another scale-up**:

```yaml
scale_up_cooldown_seconds: 45    # Wait 45s between scale-ups
```

**Purpose**: Prevent thrashing during metric spikes.

**Recommendations**:
- **Fast APIs**: 30-45s
- **Slow startup**: 60-120s
- **Cold start**: 180s+

### Scale-Down Cooldown

Wait time before **scale-down**:

```yaml
scale_down_cooldown_seconds: 300    # Wait 5 minutes
```

**Purpose**: Prevent premature scale-down during temporary dips.

**Recommendations**:
- **Stable traffic**: 180-300s
- **Variable traffic**: 600s
- **Expensive startup**: 900s+

## Custom Metrics

Scale based on application-specific metrics:

```yaml
autoscaling:
  enabled: true
  min_replicas: 2
  max_replicas: 50

  custom_metrics:
    # Active database connections
    active_connections:
      target: 100
      scale_down_threshold: 30
      query: |
        avg_over_time(pg_stat_activity_count[1m])

    # Queue depth
    queue_depth:
      target: 1000
      scale_down_threshold: 100
      query: |
        rabbitmq_queue_messages{queue="tasks"}

    # Request latency (p95)
    p95_latency_ms:
      target: 200
      scale_up_threshold: 250
      query: |
        histogram_quantile(0.95,
          rate(http_request_duration_ms_bucket[1m])
        )
```

### Custom Metric Fields

| Field | Type | Description |
|-------|------|-------------|
| `target` | number | Ideal metric value |
| `scale_down_threshold` | number | Scale down below this |
| `scale_up_threshold` | number | Scale up above this (optional) |
| `query` | string | Prometheus query |

### Custom Metric Flow

```
1. Scrape: Prometheus adapter collects metrics
2. Normalize: Convert to percentage or absolute value
3. Evaluate: Autoscaler compares to target
4. Action: Scale up/down based on thresholds
```

## Real-World Examples

### Example 1: Production API

```yaml
services:
  api:
    image: myapp/api:v2
    cpu: 2
    memory: 4Gi

    autoscaling:
      enabled: true
      min_replicas: 5        # Always 5+ for redundancy
      max_replicas: 50       # Cap at 50
      target_cpu_percent: 65 # Conservative target

      # Fast scale-up for traffic spikes
      scale_up_cooldown_seconds: 30

      # Slow scale-down to avoid yo-yo
      scale_down_cooldown_seconds: 600
```

**Result**:
- Baseline: 5 replicas ($50/month)
- Peak: 30 replicas during business hours
- Cost: ~$180/month (vs $500/month fixed 50 replicas)

### Example 2: Batch Worker

```yaml
services:
  worker:
    image: myapp/worker:v1
    cpu: 4
    memory: 8Gi

    autoscaling:
      enabled: true
      min_replicas: 0        # Scale to zero when idle
      max_replicas: 100      # Massive burst capacity

      custom_metrics:
        queue_depth:
          target: 100        # 100 jobs per worker
          query: |
            rabbitmq_queue_messages{queue="tasks"}
```

**Result**:
- Idle: 0 replicas ($0/month)
- Active: Scales to match queue (1 replica per 100 jobs)
- Cost: Pay only for processing time

### Example 3: Database Proxy

```yaml
services:
  pgbouncer:
    image: pgbouncer:latest
    cpu: 1
    memory: 2Gi

    autoscaling:
      enabled: true
      min_replicas: 2        # Always 2 for HA
      max_replicas: 10       # Limited burst

      custom_metrics:
        active_connections:
          target: 200        # 200 connections per proxy
          scale_down_threshold: 50
          query: |
            avg_over_time(pgbouncer_active_clients[1m])
```

**Result**:
- Baseline: 2 proxies (400 connections)
- Peak: 8 proxies (1,600 connections)
- Cost: ~$30/month

### Example 4: ML Inference

```yaml
services:
  inference:
    image: ml-model:v3
    cpu: 8
    memory: 16Gi
    gpu:
      count: 1
      model: nvidia-a100

    autoscaling:
      enabled: true
      min_replicas: 1        # Keep 1 warm
      max_replicas: 20       # Scale to 20 GPUs

      target_cpu_percent: 70

      custom_metrics:
        inference_queue:
          target: 50         # 50 queued requests per GPU
          query: |
            sum(rate(inference_queue_depth[1m]))
```

**Result**:
- Warm: 1 GPU ready ($2/hour)
- Peak: 10 GPUs during batch inference
- Cost: ~$1,500/month (vs $14,000/month for 20 fixed GPUs)

## Troubleshooting

### Issue: Service Not Scaling Up

**Symptom**: CPU at 100%, but replicas not increasing

**Causes**:

1. **Hit max_replicas**:
   ```yaml
   max_replicas: 5    # Increase this
   ```

2. **Scale-up cooldown active**:
   ```yaml
   scale_up_cooldown_seconds: 300    # Reduce this
   ```

3. **Provider capacity exhausted**:
   - Add more providers
   - Check region availability

**Solution**:

```bash
# Check autoscaler status
core services describe api --show-autoscaler

# View scaling events
core events list --service api --type scaling
```

### Issue: Service Scaling Too Aggressively

**Symptom**: Constantly scaling up/down (yo-yo effect)

**Cause**: Cooldowns too short

**Solution**:

```yaml
autoscaling:
  scale_up_cooldown_seconds: 60      # Increase
  scale_down_cooldown_seconds: 600   # Increase
```

### Issue: Slow Scale-Up

**Symptom**: Latency spikes before scaling kicks in

**Solutions**:

1. **Lower CPU target**:
   ```yaml
   target_cpu_percent: 60    # From 70
   ```

2. **Increase min_replicas**:
   ```yaml
   min_replicas: 5          # From 3
   ```

3. **Reduce cooldown**:
   ```yaml
   scale_up_cooldown_seconds: 30    # From 60
   ```

### Issue: Scale-to-Zero Not Working

**Symptom**: Service stays at min_replicas despite no traffic

**Cause**: min_replicas > 0

**Solution**:

```yaml
autoscaling:
  min_replicas: 0    # Enable scale-to-zero
```

**Note**: Requires container to start in <15s for best experience.

## Best Practices

### 1. Start Conservative

```yaml
# Initial deployment
autoscaling:
  min_replicas: 3
  max_replicas: 10
  target_cpu_percent: 70
```

Monitor for 1-2 weeks, then optimize.

### 2. Set Realistic Maximums

Don't set max_replicas arbitrarily high:

```yaml
# Bad: Unlimited ceiling
max_replicas: 1000    # ✗ Risk of huge costs

# Good: Realistic ceiling
max_replicas: 50      # ✓ Based on expected traffic
```

### 3. Longer Scale-Down Cooldowns

Scale down slowly to avoid instability:

```yaml
scale_up_cooldown_seconds: 45      # Fast
scale_down_cooldown_seconds: 600   # Slow (10x)
```

### 4. Monitor Autoscaler Behavior

```yaml
observability:
  metrics:
    endpoint: /metrics
  alerts:
    - name: frequent_scaling
      query: rate(autoscaling_events[5m]) > 10
      action: investigate
```

### 5. Test Under Load

Use load testing to validate:

```bash
# Load test with scaling
core load-test api --rps 1000 --duration 10m --watch-scaling
```
