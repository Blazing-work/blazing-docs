Blazing Flow is **code‑first**. There is no `flow.yaml` consumed by the SDK in the current release.

If you need persistent infrastructure, scaling, networking, and placement controls, use **Blazing Core YAML**. This page documents what you can configure **inside your Python app** today.

## App Configuration (Python)

```python
from blazing import Blazing

app = Blazing(
    api_url="https://api.blazingflow.com",
    api_token="...",
    python_version="3.11",
    dependencies=["numpy", "pandas"],
    sandbox_dependencies=["typing-extensions>=4.0.0"],
)
```

Common knobs:
- `python_version` — executor Python version
- `dependencies` — trusted executor packages (installed server‑side)
- `sandbox_dependencies` — Pyodide packages for sandboxed steps
- `sandbox_wheel_urls` — custom wheels for Pyodide

## Execution Configuration (Python)

Use decorators to configure execution:

```python
from blazing import Cron, Period

@app.step(gpu="A100", gpu_count=2)
async def train(data, services=None):
    ...

@app.workflow(schedule=Cron("0 2 * * *"))
async def nightly_report(services=None):
    ...

@app.workflow(schedule=Period(minutes=30))
async def periodic_sync(services=None):
    ...
```

## Timeouts

Use workflow execution handles to apply timeouts when waiting for results:

```python
run = await app.my_workflow(x=1)
result = await run.result(timeout=60)
```

## Persistent Infrastructure (Blazing Core)

All persistent infrastructure and scaling controls live in **Blazing Core YAML**:
- CPU/memory/GPU resources
- Autoscaling policies
- Placement/regions/providers
- Networking/ingress/egress
- Volumes and bucket mounts

If you need these, configure them in Core and keep Flow code focused on orchestration logic.
