This guide focuses on developer‑level observability that works today without exposing platform internals. Use standard Python tooling for logs, metrics, tracing, and profiling, and keep deployment concerns in Blazing Core.

## Logging

Blazing Flow captures your standard Python logs. Use structured logging so you can filter and analyze events later.

```python
import asyncio
import logging
from blazing import Blazing

logger = logging.getLogger(__name__)

async def main():
    app = Blazing()

    @app.step
    async def process_order(order_id: str, services=None):
        logger.info("processing order", extra={"order_id": order_id})
        result = await handle(order_id)
        logger.info("order completed", extra={"order_id": order_id})
        return result

    await app.publish()

async def handle(order_id: str):
    return {"order_id": order_id, "status": "completed"}

if __name__ == "__main__":
    asyncio.run(main())
```

## Metrics

For custom metrics, instrument your code with the library your team already uses (Prometheus, OpenTelemetry, StatsD, Datadog, etc.). Keep the metrics export configuration in Blazing Core or your observability stack.

Practical guidance:

- Track business metrics in steps and workflows (e.g., throughput, latency, error rate).
- Emit counters/histograms where you already handle exceptions.
- Avoid high‑cardinality labels for workflow inputs.

## Tracing

If you need traces, add OpenTelemetry (or your tracing SDK) in your workflow code. This keeps your instrumentation portable and avoids platform coupling.

```python
import asyncio
from opentelemetry import trace
from blazing import Blazing

tracer = trace.get_tracer(__name__)

async def main():
    app = Blazing()

    @app.step
    async def transform(data, services=None):
        with tracer.start_as_current_span("transform") as span:
            span.set_attribute("data_size", len(data))
            return data

    await app.publish()

if __name__ == "__main__":
    asyncio.run(main())
```

## Alerts

Set alerts in your existing monitoring system (Datadog, PagerDuty, Grafana, CloudWatch, etc.). A reliable pattern is to alert on:

- error rate (percentage of failed executions)
- p95/p99 latency
- queue backlog or input rate spikes

## Profiling

For performance investigations, use Python’s built‑in profiler locally first, then promote hot‑path fixes to production.

```bash
python -m cProfile -o profile.out workflows/workflow.py
python -m pstats profile.out
```
