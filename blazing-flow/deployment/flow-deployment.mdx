Blazing Flow deployments are **code-first**. You ship Python code that calls app.publish() and (optionally) a **Blazing Core YAML** file for persistent services and scaling. This guide focuses on production‑grade, developer‑friendly workflows without exposing internal platform details.

## Recommended Deployment Pattern

### 1. Package your workflow as a deployable script

Keep the entrypoint thin and explicit:

```python
# workflow.py
import asyncio
from blazing import Blazing

async def main():
    app = Blazing()

    @app.step
    async def process_data(data: dict, services=None):
        return {"processed": True, "data": data}

    @app.workflow
    async def data_pipeline(data: dict, services=None):
        return await process_data(data, services=services)

    await app.publish()

if __name__ == "__main__":
    asyncio.run(main())
```

### 2. Deploy from CI/CD

Your CI only needs to run the publish script with the right environment:

```yaml
# .github/workflows/deploy-flow.yml
name: Deploy Flow

on:
  push:
    branches: [main]
    paths: ["workflows/**", "src/**"]

jobs:
  deploy:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Set up Python
        uses: actions/setup-python@v5
        with:
          python-version: "3.11"

      - name: Install deps
        run: pip install -r requirements.txt

      - name: Publish
        env:
          BLAZING_API_URL: <your-api-url>
          BLAZING_API_TOKEN: <your-api-token>
        run: python workflows/workflow.py
```

### 3. Use Core YAML for persistent services and scaling

For long‑lived services, tenant isolation, and scaling policies, use **Blazing Core** YAML in the infrastructure repo and keep your Flow scripts focused on workflow logic.

## Versioning and Rollbacks

Blazing Flow versioning is driven by source control and app.publish().

Practical recommendations:

- Tag releases in Git and attach the tag/version to your workflow metadata.
- Keep the entrypoint deterministic and explicit about dependencies.
- Roll back by re‑publishing the previous tag/commit.

## Custom Executor Images

You can customize the executor image per step or workflow using Image.executor(). This allows GPU and system‑level dependencies without exposing platform internals.
