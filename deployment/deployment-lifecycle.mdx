Blazing Core Compose follows a structured deployment lifecycle: **validate → plan → apply → monitor**.

## Lifecycle Overview

```
┌─────────────┐
│   Source    │  compose.yaml
│   (YAML)    │
└──────┬──────┘
       │
       ▼
┌─────────────┐
│  Validate   │  Schema + Policy checks (CLI)
│   (Local)   │  ✓ Fast feedback (50-200ms)
└──────┬──────┘
       │
       ▼
┌─────────────┐
│    Plan     │  Show intent (CLI + Server)
│  (Preview)  │  ✓ Cost estimate, region selection
└──────┬──────┘
       │
       ▼
┌─────────────┐
│    Apply    │  Provision infrastructure (Server)
│  (Deploy)   │  ✓ Atomic execution
└──────┬──────┘
       │
       ▼
┌─────────────┐
│   Monitor   │  Track health and performance
│ (Operation) │  ✓ Metrics, logs, alerts
└─────────────┘
```

## Phase 1: Validate

Local validation of YAML configuration.

### Command

```bash
core validate compose.yaml
```

### What It Checks

1. **YAML Syntax**: Valid YAML structure
2. **Schema Compliance**: Matches Core Compose v0.2.0 schema
3. **Feature Availability**: Requested features are supported
4. **Policy Requirements**: Org-level policies are met
5. **Control Satisfaction**: Compliance controls can be satisfied

### Sample Output

```
Validating compose.yaml...

✓ YAML syntax valid
✓ Schema compliance (v0.2.0)
✓ Feature checks passed
✓ Policy requirements met
✓ Controls validated

Configuration is valid and ready to deploy.
```

### Validation Errors

```bash
$ core validate compose.yaml

  services.api.cpu: must be a number
  Found: "2GB" (string)
  Expected: 2 (number)

compose.yaml:32:9
  services.api.autoscaling.min_replicas: must be ≥ 0
  Found: -1
  Expected: 0 or greater

2 errors found. Fix these issues and try again.
```

Preview what will be deployed without actually deploying.

### Command


2. **Region Allocation**: Specific regions to be provisioned
3. **Resource Summary**: CPU, memory, GPU, storage
4. **Cost Estimate**: Monthly cost with variance
5. **Compliance Status**: Controls met/unmet
6. **Placement Rationale**: Why providers were chosen

## Phase 3: Apply

Execute the deployment plan.

### Command

```bash
core apply compose.yaml
```

### What It Does

1. **Server-Side Orchestration**: API runs placement algorithm
2. **Resource Provisioning**: Creates compute nodes, IPs, storage
3. **Network Setup**: Configures ingress/egress, TLS certificates
4. **Service Deployment**: Deploys containers across regions
5. **Health Verification**: Waits for readiness probes
6. **Audit Logging**: Records deployment details


## Deployment States

| State | Description | Actions Available |
|-------|-------------|-------------------|
| **draft** | YAML loaded but not validated | validate, plan |
| **validated** | Schema checks passed | plan, apply |
| **planning** | Computing placement | wait |
| **provisioning** | Creating infrastructure | wait, cancel |
| **deploying** | Rolling out services | wait, watch |
| **healthy** | All services running | scale, update, destroy |
| **degraded** | Some replicas unhealthy | rollback, investigate |
| **failed** | Deployment failed | retry, rollback |

## Atomic Deployments

Blazing uses **atomic deployments**: all resources succeed or all fail.

### Success Path

```
1. Provision all infrastructure  ✓
2. Deploy all services          ✓
3. Verify health checks         ✓
4. Update DNS                   ✓
→ Deployment successful
```

### Failure Path

```
1. Provision all infrastructure  ✓
2. Deploy all services          ✗  (one service fails)
3. Rollback all changes         ✓
→ Deployment failed, state restored
```

**No partial deployments**: Either everything works or nothing changes.

## Update Deployments

Modify existing deployments with `apply`:

```bash
# Edit compose.yaml
vim compose.yaml

# Apply changes
core apply compose.yaml
```

### Rolling Updates

Updates are rolled out gradually:

```
1. Deploy new replica    [new]
2. Wait for health      ✓
3. Remove old replica   [old] →
4. Repeat for all replicas
```

### Zero-Downtime Updates

Guaranteed zero downtime if:
- `min_replicas ≥ 2`
- Health checks configured
- Gradual rollout (default)

### Update Example

```bash
# Update image version
vim compose.yaml  # Change image: myapp:v2.1.0 → v2.2.0

# Apply update
$ core apply compose.yaml

  • services.api.image: myapp:v2.1.0 → myapp:v2.2.0

  • 1 replica at a time
  • Wait for health checks
  • 30 second intervals

Proceed? [y/N]: y

Updating api service...
├─ Deploying replica 1/8... healthy
├─ Deploying replica 2/8... healthy
├─ Deploying replica 3/8... healthy
...
└─ Deploying replica 8/8... healthy

✓ Update successful (0 downtime)
```

## Rollback

Revert to previous deployment:

```bash
# View deployment history
core deployments list

# Rollback to previous
core rollback

# Rollback to specific deployment
core rollback --to df-2025-12-03-00300
```

### Automatic Rollback

Blazing automatically rolls back if:
- Health checks fail
- Deployment timeout exceeded
- Critical error during apply

## Deployment Comparison

Compare two deployments:

```bash
# Compare with previous

# Compare with specific deployment
core diff --compare df-2025-12-03-00300

# Show only service changes
core diff --services-only
```

## Best Practices

### 1. Always Validate First

```bash
core validate compose.yaml && core plan compose.yaml && core apply compose.yaml
```

### 2. Review Plans Carefully

Check:
- Cost estimates
- Region selection
- Compliance status
- Resource allocation

### 3. Use Staged Rollouts

```bash
# Deploy to staging first
core apply --project staging

# Verify
core services list --project staging

# Deploy to production
core apply --project production
```

### 4. Monitor After Deploy

```bash
core apply compose.yaml && core logs api --follow
```

Watch for errors in the first 5-10 minutes.

### 5. Tag Deployments

```bash
core apply --tag "release-v2.2.0"
core apply --tag "hotfix-security"
```

Makes rollback easier:

```bash
core rollback --to-tag "release-v2.1.0"
```
